01الكتالوج الكامل · ١٢ ايجنت في طبقتين

طبقة التنسيق تدير الجلسة وتشغّل الموجات. طبقة المتخصّصين تُنتج البطاقات (EntityCards) لكل جهة حكومية. الـ LLM المستخدم في كل ايجنت ذكي هو claude-sonnet-4-6 — موديل واحد عبر الـ Anthropic SDK، مع backup key للفشل عند 429. الايجنتات الحتمية (deterministic) ما تستدعي LLM إطلاقاً.

02AgentBus — كيف يتواصلون فعلياً

الـ AgentBus هو قلب الـ A2A. ليس logger UI — كل ايجنت عنده inbox مستقل، والـ outbox تنطلق رسائله للـ inboxes الأخرى لمّا الـ wave يخلص. الايجنت الجاي في الـ wave التالي يفتح inboxه، يقرأ الرسائل، ويبني سلوكه عليها (مثلاً municipality يضيف رخصة المطبخ فقط لو وصلتها رسالة من civil_defense تقول hasKitchen: true).

أنواع الرسائل (AgentMessage.type)

النوع متى يُستخدم مثال حقيقي
dependency handoff بين ايجنتين متسلسلين — الجاي ينتظره civil_defense → municipality: { safetyCertReady: true, hasKitchen: true }

update تحديث تنظيمي من research موجّه لمتخصص research → sfda: «اشتراطات جديدة لتغليف اللحوم 2026-Q2»
warning تنبيه يحتاجه ايجنت آخر في قراره zatca → mci: «المنشأة تجاوزت حد الـ VAT»
ack تأكيد استلام أو إكمال analysis → report: «اكتملت ٤ ثغرات حرجة»

قواعد التسليم

  • to: 'ALL' = broadcast لكل ايجنت مسجَّل في الـ run، باستثناء المرسل نفسه.
  • to: AgentId = توصيل مباشر للـ inbox المعني. لو الـ id مش مسجَّل في هذا الـ run، الرسالة تُسقَط من الـ bus لكن تبقى في الـ log.
  • الترتيب محفوظ — الـ bus يحتفظ بـ seq تصاعدي لكل رسالة، والـ UI يعتمد عليه لاكتشاف الجديد بدون refetch كامل.
  • blocked retry — لو ايجنت رجع status: 'blocked' (الـ inbox ناقص)، الـ scheduler يعيده في موجة لاحقة حتى ٣ مرات. أكثر من ذلك يعتبر deadlocked.
  • كشف الـ deadlock — أي wave ما فيها agent ready يرمي خطأ بدلاً من الانتظار اللانهائي. MAX_WAVES = 16 سقف صلب.

الـ system prompt — قواعد لا يكسرها الموديل

// src/agents/specialists/civil-defense-agent.ts
أنت متخصّص الدفاع المدني السعودي. مهمّتك تحديد متطلبات السلامة لمحلّ صغير
ثم صياغة شهادة السلامة المطلوبة وتفسيرها بعربية واضحة.

القواعد:
- لا تخترع أرقاماً. كل عدد طفايات أو تكلفة أو متطلب يجي من الـ tools.
- استخدم get_shop_summary أولاً لفهم المحل.
- استخدم list_safety_requirements للقاعدة، ثم
  calculate_extinguisher_count لو في الإجابة عدد ناقص.
- لو في رسالة من research في الـ inbox فيها تحديث تنظيمي،
  أضفه كـ "جديد —" في requirements.
- في outbox: أرسل dependency message لـ municipality فيها
  { safetyCertReady, hasKitchen, estimatedTimeAr }.

المخرج (EntityInfo + outbox)

  • data: explainAr (شرح عربي)، estimatedCostSar { min, max }، estimatedTimeAr، requirements[]، criticalWarningAr، renewalMonths.
  • outbox: رسالة dependency لـ municipality. payload يحدد لو في مطبخ (محل مطعم/كوفي) — البلدية تضيف بناءً عليها رخصة مطبخ تجاري تلقائياً.
  • trace: عدد التكرارات، tokens المستهلكة، latency، النص النهائي. تُحفظ في project.agentTraces ويعرضها الـ UI في الداشبورد.
fallback الحتمي: لو فشل النداء (لا يوجد API key، rate limit، 5xx، JSON غير صالح بعد ٦ تكرارات) — الايجنت يستخدم منطق محلي مشتق من src/knowledge/entities ويرجع نفس شكل الـ EntityInfo. الخدمة لا تتوقّف لأن الموديل تأخّر.

07مراقبة + خطأ + مرونة

التتبع (Trace)

كل ايجنت LLM يخزّن trace كامل في project.agentTraces[agentId]:

  • iterations[] — كل تكرار للـ tool loop (المدخل، الأداة المُستدعاة، النتيجة).
  • totalInputTokens / totalOutputTokens — استهلاك tokens.
  • totalLatencyMs — زمن النداء الكلي.
  • mode: 'live' (Claude نجح) أو 'fallback' (استخدمنا الحتمي مع fallbackReason).

الواجهة في /project/[id] تعرض هذه التتبّعات في panel لكل ايجنت — الحكام يقدرون يفتحون أي بطاقة ويشوفون وش بالضبط طلبه الموديل.

سياسات الفشل (Resilience)

الفشلالاستجابة
Anthropic 429 (rate limit) تحويل تلقائي لـ BACKUP_ANTHROPIC_API_KEY لو موجود (src/lib/claude.ts)
JSON غير صالح من الموديل parseAgentJson يلتقطه، الـ specialist يستدعي fallback()
tool loop تجاوز ٦ تكرارات قطع وفشل آمن → fallback
web_search رجع شيء غير array research يستخدم FALLBACK_UPDATES (قائمة معتمدة من مواقع gov.sa)
ايجنت مسدود (inbox ناقص) retry حتى ٣ مرات، ثم error: blocked past N retries
Vercel كاد يقتل الـ lambda waitUntil + flushBus يحفظون آخر دفعة قبل القتل
Anthropic key مفقود كل ايجنت LLM يدخل fallback() فوراً — التطبيق لا يفشل

حدود واضحة (Limits)

  • MAX_WAVES = 16 — كشف dependency cycles.
  • MAX_BLOCKED_RETRIES = 3 — لا انتظار لانهائي.
  • maxIterations = 6 لكل tool loop — لا حلقات لانهائية.
  • maxTokens = 1500 لكل نداء specialist، 2000 للبحث.

08FAQ — توقّع أسئلة الحكام

هل الـ A2A فعلي ولا decoration؟
فعلي. لو شلت رسالة civil_defense → municipality ما تظهر «رخصة المطبخ التجاري» في تقرير المطعم. المصدر: src/agents/specialists/civil-defense-agent.ts سطر ٦٥، والاستهلاك في municipality-agent.ts blockedReason.
ليش ما تستخدمون LangChain أو CrewAI أو AutoGen؟
لأن المتطلبات معروفة ومحدودة (٧ متخصصين، ٤ أنواع رسائل، wave scheduler). كتابتها مباشرة بـ TypeScript أعطتنا: types صريحة، traces مخصّصة، لا dependencies إضافية، وقدرة كاملة على debugging الـ deadlocks. الكود كله أقل من ٤٠٠ سطر للـ runtime + bus.
كيف تضمنون أن الموديل ما يهلوس أرقاماً (مثل عدد الطفايات)؟
الـ system prompt يقول صراحة: «لا تخترع أرقاماً. استخدم الـ tools». الـ tools نفسها دوال TypeScript محلية بأرقام مأخوذة من knowledge base (src/knowledge/entities). الموديل يستدعيها — ما يحسبها. لو تجاوز الـ instruction (نادر)، الـ parseAgentJson يطّبق schema validation، وأي خرق يدخل في الـ fallback الحتمي.
ليش Sonnet 4.6 وليس Haiku أو Opus؟
Sonnet 4.6 هو أفضل model من ناحية coding/reasoning مع سعر معقول و latency جيدة. Haiku أسرع لكن دقّته في tool-use loops أقل في الاختبارات اللي سوّيناها. Opus 4.7 مخصّص للـ planning العميق فقط — هنا الـ pipeline متجزّأ على ٧ متخصصين، فتنوع المهام بسيط نسبياً وهيكل الـ prompts صارم. النتيجة: ~3.2 ثانية latency متوسط لكل specialist.
وش يصير لو انقطع Claude API منتصف الـ run؟
كل specialist يلتقط الاستثناء ويستدعي fallback() — منطق حتمي يرجع نفس شكل الـ EntityInfo ويعلِّم الـ trace كـ 'fallback'. الـ run يكمل، الـ UI يعرض البطاقات مع علامة «منطق احتياطي» (لو توسّعنا الـ UI).
كيف يعرف الـ scheduler أن الايجنت مسدود؟
الـ specialist يرجع { status: 'blocked', reason: '...' } عند فحص الـ blockedReason(context, inbox). مثلاً municipality يرجع blocked لو ما شاف safetyCertReady: true من civil_defense. الـ scheduler يبقيه في pending ويعيد محاولته بعد كل wave جديدة.
هل الـ research فعلاً يبحث على الويب أو fakes the data؟
يستخدم أداة Anthropic الرسمية web_search_20250305 ويفلتر النتائج ليبقي بس روابط gov.sa + إعلام موثوق. لو الـ tool ما توفّر في إصدار الـ SDK أو رجع schema غريب، يسقط على FALLBACK_UPDATES (تحديثات مدوّنة بتواريخ ومصادر حقيقية).
المعمارية scalable لو زدنا ايجنتات تانية (مثلاً وزارة النقل)؟
نعم. تضيف class جديد يرث من LlmSpecialistAgent، تحدد id + dependencies، تكتب system/user prompts + tools + parseOutput + fallback، تسجّله في specialists/index.ts. الـ scheduler يلتقطه تلقائياً ويرتّبه في الموجة المناسبة.
وش الفرق بين الـ activities والـ messages؟
activities = حالة الايجنت نفسه (started/working/completed/error) للعرض في الـ UI. messages = حمولة فعلية بين ايجنتين عبر الـ bus، تحوي type و payload. الاثنان يسلسَلان معاً في الـ timeline بعمود seq مشترك.
هل تخزّنون كلام الموديل (PII / privacy)؟
نخزّن trace لكل ايجنت (iterations + tokens + final text) في Supabase ضمن project.agentTraces — البيانات تخص المحل نفسه (نشاط، مساحة، عدد عمالة) وليست بيانات شخصية. الإيميل اختياري، يُخزَّن مفصول في project.email ولا يُمرّر للـ LLM في الـ prompts.
